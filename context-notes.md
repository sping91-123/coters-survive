# 작업 맥락 메모

## 2026-05-13 6시간 반복 점검 41차 통합 스모크 회귀 방지.

- `smoke:all`이 서버 의존 검증 전에 `dev:clean`과 헬스체크 대기를 수행하도록 바뀌었지만, 이 흐름 자체가 빠지면 같은 캐시 문제가 다시 생길 수 있습니다.
- 운영 스모크가 `scripts/smoke-all.mjs` 안의 `dev:clean`, `waitForDevServer`, `/api/health` 대기 흐름을 감시하게 두면 검증 체계의 품질을 유지하기 쉽습니다.

## 2026-05-13 6시간 반복 점검 40차 통합 스모크 개발 서버 자동 복구.

- `smoke:all` 실패는 결제 라우트 로직 자체보다 오래된 `.next` 개발 서버 캐시가 깨진 chunk를 참조한 것이 원인이었습니다.
- 사용자가 반복적으로 겪은 “텍스트만 뜸”, “페이지가 안 뜸” 문제와 같은 계열이므로, 통합 검증은 서버 의존 테스트 직전에 개발 서버를 깨끗하게 재시작해야 신뢰할 수 있습니다.
- `smoke:all`이 운영·모바일·결제 같은 정적 검증을 먼저 끝낸 뒤, API와 라우트 검증 전 `dev:clean`을 실행하도록 정리하는 방향이 맞습니다.

## 2026-05-13 6시간 반복 점검 39차 개발 서버 복구 스크립트 재시도.

- 빌드와 개발 서버가 겹치면 Windows에서 `.next` 내부 파일이 잠깐 잠겨 `Remove-Item`이 실패할 수 있습니다.
- 사용자는 이 상황을 “페이지가 텍스트로만 뜬다”로 체감하므로, 복구 명령이 한 번 실패하고 끝나는 것은 운영 경험에 좋지 않습니다.
- `.next` 삭제를 짧은 간격으로 재시도하게 만들어 개발 서버 복구 명령을 더 튼튼하게 둡니다.

## 2026-05-13 6시간 반복 점검 38차 관심코인 스캔 입력 검증.

- 관심코인 스캔은 사용자가 저장한 코인 목록을 서버에서 다시 판독하는 흐름입니다.
- 잘못된 심볼이 들어왔을 때 조용히 빈 결과를 반환하면, 저장값 문제인지 시장에 후보가 없는지 구분하기 어렵습니다.
- 다른 공개 API처럼 잘못된 입력은 400으로 거절해 UI 버그와 비정상 요청을 빨리 발견하게 합니다.

- 검색엔진과 공유 메타 기준에서도 글로벌 레이더를 코인 레이더와 같은 1차 진입 화면으로 다루도록 우선순위를 정렬합니다.

- 로그인 토큰이 없어도 플랜과 금액 검증은 인증 검사보다 먼저 실행되므로, 라우트 스모크에서 불일치 플랜 요청이 400으로 막히는지 확인할 수 있습니다.
- 이 케이스를 라우트 스모크에 넣어 실제 서버 동작 기준의 결제 회귀 방지 장치를 하나 더 둡니다.

## 2026-05-13 6시간 반복 점검 35차 결제 승인 플랜 무결성.

- 결제 승인 API는 주문번호(`cr_{planId}_...`)와 요청 body의 `planId`를 함께 받을 수 있습니다.
- 지금처럼 같은 가격의 Coin Pro 월간과 Global Pro 월간이 있으면, body의 `planId`를 먼저 믿는 방식은 결제한 시장과 다른 시장 권한을 열 여지를 만듭니다.
- 결제 성공 후 권한은 주문번호에 들어간 플랜을 기준으로 확정하고, body에 별도 `planId`가 들어왔는데 주문번호와 다르면 요청을 거절하는 편이 안전합니다.

## 2026-05-13 6시간 반복 점검 34차 배포 URL fallback 정리.

- sitemap은 Vercel 배포 URL fallback을 갖고 있었지만 layout metadata와 robots는 같은 정책이 아니었습니다.
- 도메인 환경변수를 깜빡해도 Vercel 배포에서는 공유 메타와 robots가 로컬 주소로 고정되지 않도록 공통 URL 유틸로 합칩니다.
- 헬스체크도 `NEXT_PUBLIC_SITE_URL` 또는 `VERCEL_URL` 둘 중 하나가 있으면 사이트 URL 준비로 인정합니다.

## 2026-05-13 자동 매크로 캘린더 1차.

- 기존 매크로 일정은 `src/data/macroEvents.ts`의 정적 데이터라 발표 후 실제값이 자동으로 들어오지 않았다. 사용자가 지적한 대로 이 상태로는 “매일 켜는 앱”의 핵심 기능으로 보기 어렵다.
- 최선의 1차 구조는 Trading Economics 경제 캘린더 API를 우선 사용하고, API 키가 없거나 장애가 날 때 기존 수동 데이터로 fallback하는 방식이다.
- Trading Economics 문서상 캘린더 응답에는 Actual, Previous, Forecast, Importance, LastUpdate, SourceURL이 포함되어 발표 전후 자동 갱신에 필요한 필드가 맞다.
- 다만 외부 유료 API가 바로 준비되지 않아도 앱이 죽으면 안 되므로, 화면과 헬스체크는 자동 연결 여부를 표시하되 사용자 화면은 계속 정상 동작하게 한다.

## 2026-05-13 무료 공식 매크로 자동 갱신 전환.

- 사용자가 지적한 대로 Trading Economics는 유료 API라서 출시 기본 인프라로 강제하면 안 된다.
- 1차 해결은 BLS 공개 API를 기본값으로 사용해 CPI와 PPI 실제값을 자동 계산하고, 나머지 일정과 예상치는 기존 운영 백업 캘린더를 합치는 방식이다.
- Trading Economics는 더 넓은 캘린더와 예상치 자동화를 원할 때 붙이는 선택형 유료 보강 옵션으로만 남긴다.
- `/api/health`의 유료 출시 준비 판단도 유료 매크로 키를 필수로 요구하지 않도록 바꾼다. 무료 공식 매크로 제공자가 살아 있으면 매크로 자동화는 충족된 것으로 본다.
## 2026-05-14 뉴스 한국어화와 무료/Pro 가치 분리

- 사용자가 뉴스 화면에서 영어가 많이 보인다고 느낀 원인은 두 가지다. 첫째, RSS 제목 번역 실패 시 영어 제목이 그대로 `translatedTitle`에 들어간다. 둘째, 카드 UI가 한국어 제목 아래에 원문 영어 제목을 다시 보여준다.
- 뉴스 화면의 기본 경험은 한국어 브리핑이어야 한다. 원문은 신뢰 확인용 링크로만 남기고, 카드 제목과 AI 요약은 한국어로 보여주는 방향이 맞다.
- 무료 플랜은 제품 감각을 확인하는 체험판이어야 한다. 무료로도 충분히 쓸 수 있으면 월 구독 이유가 약해지므로, 반복 감시, 관심종목, 알림, AI 브리핑은 Pro의 핵심 가치로 분명히 보여줘야 한다.
- 이번 작업에서는 결제 서버 권한 전체 개편까지는 건드리지 않고, 사용자에게 보이는 무료/Pro 가치와 일일 사용 기준을 먼저 정리한다.
- 구현 후 확인 결과, 뉴스 카드에는 원문 영어 제목을 다시 표시하지 않는다. API는 Groq가 가능하면 제목을 한국어로 의역하고, 실패해도 규칙 기반 한국어 이슈 제목으로 대체한다.
- 빌드 중 켜져 있던 개발 서버가 `.next` 청크 꼬임으로 500을 냈고, `npm run dev:clean`으로 복구했다. 이후 `smoke:api`, `smoke:routes`가 통과했다.
